~/Inter-Act/02-imagine

Imagine

A base in space

Multiple actors, all doing their designated tasks. Robots that move material, robots that build, robots that grow food, rovers that scout the surface. Different jobs, different expertise, all working in the same place at the same time.

With the current approach, you would need one central unit to handle and combine every behaviour. That works fine on a small or mid-size scenario. The moment you add more units, it falls over. One brain can't keep up with the compute that many actors demand.

  • Different actors,
    different tasks
  • Demanding environment,
    outside influences
  • Round-trip latency:
    ~2-3 seconds
  • Spare parts arrive
    in days, not months

So what if the brain isn't one central unit at all?

~/Inter-Act/05-architecture

Architecture

Backend is the source of truth

state · health · LLM orchestration (demo-runner)
Backend is the source of truth. Every container reports its state and health to it, and it's also the gatekeeper for the LLM through the QwenBrain service. Demo-runner lives in its own container next door, so heavy orchestration work doesn't tie up the API thread pool. It only talks to arms and assets directly while a demo is running.

~/Inter-Act/08-future

Future work

Where INTER.act is heading

Two tracks, running in parallel. One makes the system easier to author. The other gives every station its own brain.

▸ UI · click & create

Node-RED-style builder

  • Drag-and-drop arms and assets, so nobody has to hand-edit YAML or code.
  • 2-D routes instead of 1-D linear chains.
  • Set each station's speed, capacity, and max load right in the UI.
  • A config panel for live edits.
▸ AI · agent-driven

An LLM brain per station

  • Qwen 2.5 (0.5B) on Ollama is already wired in, hitting 88.9% on routing.
  • Each arm gets its own brain that thinks locally.
  • Arm-to-arm and arm-to-asset negotiation; per-module constraint discovery.
  • Negative tests for when the brain hallucinates.
